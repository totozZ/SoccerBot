## 一、系统架构与设计理念

### Q1：你们的系统做了什么？

**参考作答**：一台真实的 FRC机器人作为"输入设备"，按下发射键触发 Unity 虚拟引擎生成一颗足球，虚拟队友和对手按照预设的战术剧本自动推演传球、拦截、射门的完整过程，最终给出评分。整个过程可以在 Meta Quest 3S VR 头显里以第一人称沉浸式观看。本质上是「真硬件触发，虚拟世界推演」的 AR 训练沙盒。

---

### Q2：为什么不用纯虚拟仿真，而要接一台真机器人？

**参考作答**：
- **触感真实**：真机器人的物理发射动作（炮台瞄准、飞轮旋转）无法被键盘鼠标替代。保留真实硬件作为"触发器"，让训练者感受到物理世界的因果链条——我按下发射键，虚拟世界才启动。
- **增量验证**：系统架构上，数据源（IDataSource）是可插拔的。现在用 FakeDataGenerator 独立运行调试，后期接入 NetworkTables 真实数据时，上层所有逻辑（GameManager → ScenarioPlayer → ScorePanel）一行代码都不用改。真机器人和假数据的切换只是一行配置。

---

### Q3：你们的系统架构是怎么设计的？各模块之间怎么通信？

**参考作答**：
系统采用**事件总线 + 依赖注入**的松耦合架构。核心是 `GameManager` 单例，它持有 `IDataSource` 接口——无论是 FakeDataGenerator（调试模式）还是 NTManager（真机模式），只要实现了 `IDataSource`，就能无缝切换数据源。

跨模块通信通过 4 个 C# 事件完成，**不做直接引用**：
- `OnRobotUpdated` → 状态面板、相机跟随
- `OnShooterUpdated` → 机器人炮台动画
- `OnShotFired` → 剧本触发器、球生成
- `OnConnectionChanged` → UI 连接状态

任何模块只需要订阅自己关心的事件，不需要知道数据从哪里来。这套架构使我们在 14 天内完成了 11 个模块（Core / Robot / Ball / Player / Camera / Scenario / Flow / UI / Field / Effects / XR）的并行开发，且模块间从未出现循环依赖。

---

### Q4：你们说自己是"沙盒"而不是"游戏"，区别在哪？

**参考作答**：
- **目标不同**：游戏的目的是娱乐，沙盒的目的是训练。我们提供的是可量化的反馈闭环——射门力度、传球方向、剧本结果都会转化为评分和改进建议。
- **可替换性**：沙盒的每个模块都是可替换的——剧本可以从预设 3 个扩展到 100 个，数据源可以从 FakeData 切换到 NT，AI 决策可以从固定剧本升级为强化学习。这些替换不影响其他模块。
- **开放性**：教练可以自定义剧本参数（传球路线、对手拦截半径、评分标准），这是训练工具的特征，不是游戏的固定关卡。

## 二、通信与数据链路

### Q5：真机器人和 Unity 之间怎么通信？网络延迟怎么处理？

**参考作答**：
使用 **NetworkTables v4**（WPILib 标准通信协议），通过 RoboRIO 主控器的以太网/Wi-Fi 接口与 PC 通信。NT 是一个分布式的键值存储系统，机器人端以 50Hz 频率推送位姿数据到 `/SmartDashboard/robot/pose/*` 等主题，Unity 端订阅对应主题即可获取实时数据。

延迟方面：
- 局域网内 NT 典型延迟 < 10ms，对足球机器人训练完全可接受
- 当前阶段（演示期）使用 FakeDataGenerator 替代真实 NT，确保无机器人也能独立运行
- 架构上 `IDataSource` 接口已经预留了 `IsConnected` 状态和自动重连机制（每 2 秒重试），真机接入时直接填入 RoboRIO IP 即可

### Q7：BS-BT91 智能足球的 BLE 通信是怎么设计的？

**参考作答**：
BS-BT91 是一款 BLE 5.0 九轴 IMU 姿态传感器（重量 9g，内置 Cortex-M4 + 卡尔曼滤波，姿态精度 0.2°），我们计划将其封装在 3D 打印的足球外壳内，通过蓝牙向 Unity 实时传输足球的旋转姿态（四元数 + 欧拉角 + 角速度）。

通信架构：
```
真足球 (BS-BT91 BLE)
    │ BLE 5.0 Notify (默认 10Hz，最高 200Hz)
    ▼
ISmartBallSource 接口
    ├─ MockSmartBallSource   ← PC 调试（鼠标拖拽模拟旋转）
    └─ BleSmartBallSource    ← Quest/Android BLE 直连
        │
        ▼
SmartBallController
    └─ Quaternion.Slerp 平滑 → Transform.rotation
```

关键设计决策：
- **只传旋转，不传位置**：IMU 天然无法输出绝对位置。位置跟踪（UWB/视觉）不在本期范围，球的位置由 Unity 虚拟场景固定（训练场中心），只有旋转是真实数据驱动的。这样足球在 VR 中的旋转和物理世界的真球完全同步，形成"数字孪生"的视觉锚点。
- **Android BLE 插件**：Quest 本质是 Android 设备，可以直接调用 Android BLE API（`AndroidJavaObject`），不依赖第三方插件。代码骨架 `BleSmartBallSource.cs` 已写好，只差数据帧解析器。

## 三、Unity 模拟与物理

### Q8：虚拟球的物理是怎么实现的？是真物理引擎还是自己算的？

**参考作答**：
两部分物理分开处理：

**抛物线计算（自研）**：球的飞行轨迹使用经典抛体运动公式手动计算——给定初速度矢量 + 重力加速度（9.81 m/s²），每帧步进 dt = 0.05s 生成轨迹点列表。这样做的原因是：
1. Unity 内置 PhysX 的刚体模拟存在浮点累积误差，不利于精确的轨迹预测
2. 我们需要在球发射之前就生成完整的**预测轨迹线**（通过 `TrajectoryRenderer` + `LineRenderer` 可视化），Unity 物理引擎做不到"预计算"
3. 剧本回放时需要精确复现完全相同的轨迹，确定性计算比物理引擎更可靠

**碰撞检测（Unity PhysX）**：球飞行过程中的碰撞（碰到球员/地面/球门框）由 Unity 的 `Rigidbody` + `Collider` 处理。剧本回放模式下切换为运动学模式（`isKinematic = true`），球的运动完全由关键帧动画驱动，不参与物理模拟。

---

### Q9：球的轨迹预测线是怎么画出来的？

**参考作答**：
`TrajectoryRenderer.cs` 监听 `GameManager.OnShotFired` 事件，在发射瞬间根据发射器角度、飞轮转速、机器人朝向计算出初速度矢量。然后用匀加速公式 `p(t) = p₀ + v₀·t + ½g·t²` 从 t=0 步进到球落地，每 0.05 秒采样一个点，存入 `LineRenderer` 的 position 数组。

轨迹线的颜色渐变（绿→黄→红）表示速度衰减，给训练者直观的力度反馈。这条线的长度和弧度直接反映了发射参数是否合理（角度过高→抛物线太陡、速度不够→飞不到球门）。

---

### Q10：球的发射起点是怎么确定的？跟随机器人移动吗？

**参考作答**：
是的。`MatchFlowController` 在每个 Setup 阶段，通过 `_ball.AttachTo(_robotTransform, _ballOffsetRobot)` 将球吸附到机器人的发射口位置，偏移量是 `(0, 1.0, 0.4)` 即发射口上方 1 米、前方 0.4 米。

在 Possession 阶段（玩家持球），球从机器人脚下通过抛物线飞向玩家，`_passApex = 2.2m` 控制传球弧线高度。传球过程中球不跟随任何物体，飞行路径完全由关键帧插值驱动。

未来如果真机器人移动，球吸附位置会自动跟随，因为 `_ball.AttachTo` 绑定的父 Transform 会随机器人位姿更新而同步移动。

## 四、剧本系统与 AI

### Q11：你们的"AI 剧本推演"是真正的 AI 吗？

**参考作答**：
坦诚地说，当前阶段的"AI"是指**预设剧本驱动的自动推演**，不是深度学习或强化学习。

但我们认为这对演示阶段是合理的：
- 剧本不是"放视频"——每个剧本包含多组关键帧（球路径 + 队友路径 + 对手路径），`ScenarioPlayer` 在运行时做线性插值，对手和队友的位置会根据球的飞行实时调整朝向（对手视线跟踪球、队友跑位接应），有一定的动态表现力。
- 剧本选择有随机性：`ScenarioTrigger` 从 3 个剧本中随机抽取，或根据玩家蓄力力度（≥70%→进球、40-70%→射偏、<40%→拦截）动态选择，不是固定顺序。
- 架构上已经为真 AI 留好了接口：`ScenarioPlayer` 只负责播放关键帧，未来替换为 ML-Agents 决策模块时，只需改 `ScenarioTrigger` 的选剧本逻辑，播放层完全不动。

在 PPT 的"未来展望"部分我们会明确说明：下一步计划用 Unity ML-Agents + PPO/SAC 强化学习替代预设剧本，训练环境已就绪，奖励函数设计（进球+1/拦截-1/传球成功率）也已明确。

---

### Q12：三个剧本分别是什么？怎么实现的？

**参考作答**：
三个剧本以 ScriptableObject（`.asset` 文件）的形式存储在 `Assets/Scenarios/` 目录下：

| 剧本 | 结果 | 评分 | 关键帧描述 |
|------|------|------|-----------|
| `ScoreSuccess` | 队友射门进球 | 100 分 | 球沿弧线飞入球门左上死角，对手拦截失败，队友举手庆祝 |
| `Intercepted` | 对手拦截成功 | 30 分 | 对手提前预判传球路线，横向移动截断球路，球被踢飞 |
| `ShotMissed` | 队友射偏 | 50 分 | 球擦着门柱外侧飞出底线 |

每个剧本的数据结构（`Scenario.cs`）：
- `ScenarioOutcome outcome`：结果类型枚举
- `int finalScore`：评分
- `float duration`：剧本总时长（秒）
- `Waypoint[] ballPath`：球的关键帧路径
- `Waypoint[] teammatePath`：队友关键帧路径
- `Waypoint[] opponentPath`：对手关键帧路径

`ScenarioPlayer` 在播放时按时间线性插值每个 Waypoint 的 position + rotation，驱动场景中各个 GameObject 的运动。`ScenarioFactory.cs` 编辑器工具可以一键生成新剧本的 `.asset` 文件。

---

### Q13：剧本系统的关键帧插值是用什么算法？如何处理时间同步？

**参考作答**：
使用**分段线性插值**（Lerp）。给定当前播放时间 `t`，在关键帧数组中找到相邻的两个 Waypoint `(t₁, pos₁)` 和 `(t₂, pos₂)`，计算 `α = (t - t₁) / (t₂ - t₁)`，然后 `pos = Vector3.Lerp(pos₁, pos₂, α)`。

选择线性插值而非贝塞尔曲线的原因是：
1. 可预测性强：美术/策划在编辑器中看到的关键帧位置，播放时球一定会经过
2. 性能开销极小：每帧只需一次二分查找 + 一次 Lerp，不影响 VR 帧率
3. 足够灵活：可以通过增加关键帧密度来近似任意曲线，2-3 个关键帧就足够模拟传球弧线

时间同步由 `ScenarioPlayer` 的 `Time.time` 驱动，不受帧率波动影响。回放阶段通过 `Time.timeScale = 0.3f` 实现慢动作效果，所有插值自动跟随放慢。

## 五、VR 部署与性能优化

### Q14：Meta Quest 3S 的 APK 是怎么构建和部署的？

**参考作答**：
完整流程三步走：

**1. 平台切换**：Unity Build Settings → 切换为 Android，Texture Compression 设为 ASTC。
**2. APK 构建**：`BuildAndroid.cs` 编辑器脚本调用 `BuildPipeline.BuildPlayer()`，自动设置：
- `XR Interaction Toolkit` + `Oculus XR Plugin` (v4.5.4)
- `Universal Render Pipeline` (URP 17.4)
- Android Min API Level 32（Quest 3S 要求）
- 剥离不必要的引擎模块（Physics 3D / Terrain / UnityEngine.UI 等）
**3. Sideload 部署**：通过 Meta Quest Developer Hub (MQDH) 的 `adb install` 将 APK 直装到 Quest 3S，或使用 `quest-build` skill 一键完成「切平台→Build→adb install→adb launch」。

当前已成功部署并启动，VR 内可见完整球场、机器人、虚拟球员和剧本演算过程。

### Q15：VR 的性能优化目标是什么？目前达到了吗？

**参考作答**：

| 指标 | 当前（估算） | 目标 | 优化手段 |
|------|-------------|------|----------|
| 帧率 | ~45-55 fps | **稳定 72 fps** | LOD / Shader 降级 / 粒子预算 |
| Draw Call | ~300-400 | **< 200** | 静态合批 / 材质合并 |
| SetPass Call | ~150 | **< 80** | URP Simple Lit 替换 Lit |
| 三角面数 | ~50K | **< 30K** | 3 级 LOD |
| APK 体积 | ~180MB | **< 150MB** | ASTC 6×6 压缩 + 引擎模块剥离 |

优化手段包括：
- **静态合批**：球场地面 / 球门 / 边线标记为 Static
- **纹理压缩**：RGBA32 → ASTC 6×6，人物纹理降分辨率
- **LOD**：海绵宝宝模型 3 级 LOD（VR 中距离远时用低面数）
- **粒子预算**：OutcomeFx 粒子从 50 降到 20，关闭 collision
- **阴影**：Quest 端阴影距离 10m → 5m，分辨率 1024 → 512
- **对象池**：球 / 粒子 / UI 弹窗用对象池替代 Instantiate/Destroy

当前 P9（性能优化）尚未启动，预计 1-2 天完成。演示视频会优先录制 PC 端 120fps+ 画面，VR 端作为"已跑通"的亮点展示。

### Q16：PC 端和 VR 端用的是同一份代码吗？怎么适配双端差异？

**参考作答**：
是的，同一份 Unity 工程、同一份 C# 代码，通过**条件编译 + 运行时检测**适配双端：

**条件编译**：
```csharp
#if UNITY_ANDROID && !UNITY_EDITOR
    // Quest 端：启用 VR 手柄输入
#else
    // PC 端：启用鼠标键盘 + 自由视角
#endif
```

**运行时检测**：
- `XRSetup.cs` 在 `Start()` 中检测 `XRSettings.enabled`，PC 端激活 `PCCameraController`（右键拖拽 + WASD），VR 端激活 XR Origin + TrackedPoseDriver
- `FPSPlayerController.cs` 的输入绑定同时支持 PC（Mouse LMB + RMB + WASD）和 VR（Quest 手柄 trigger / thumbstick / A 键），通过 Unity Input System 的多源绑定实现
- UI Canvas：PC 端用 `ScreenSpaceOverlay`，VR 端由 `XRSetup` 在运行时将所有 Canvas 转为 `WorldSpace` 模式

**关键差异**：
- PC 端有自由视角（右键拖拽 + 数字键 1/2/3 切机位），VR 端视角完全由头显追踪
- PC 端力度条用鼠标左键长按蓄力，VR 端用 Quest 手柄扳机键
- 球的大小在 VR 端做了 0.5× 世界缩放，PC 端保持原始比例



### Q：开发过程中遇到的最大技术挑战是什么？怎么解决的？

**参考作答**：
最具挑战性的是 **VR 端的 Canvas UI 适配**。Unity 的 UI 系统默认在 PC 端使用 `ScreenSpaceOverlay` 模式，但在 Quest VR 中必须转为 `WorldSpace` 模式才能正确渲染。问题出在：

1. 主菜单面板（`MainMenuPanel`）是运行时动态创建的，在 `XRSetup` 遍历场景中所有 Canvas 并转换模式**之后**才被实例化，导致它停留在 `ScreenSpaceOverlay` 模式
2. 症状：VR 中主菜单完全不可见，但按 A 键仍能触发游戏逻辑（因为 Quest 输入绑定仍在工作）

解决方案：在 `MainMenuPanel.EnsureRuntimeUi()` 中增加 `#if UNITY_ANDROID` 条件编译，检测到 Android 平台时直接用 `WorldSpace` 模式创建 Canvas。或者让 `XRSetup` 延迟一帧再执行 Canvas 转换，确保动态创建的 Canvas 也能被覆盖。

这个 bug 让我们深刻理解了 VR UI 的生命周期管理——必须在 Canvas 创建的同时就确定它的渲染模式，而不是事后补救。

## 七、智能足球（P11）

### Q20：BS-BT91 智能足球是什么？怎么和你们的系统结合？

**参考作答**：
BS-BT91 是博森泰克出品的一款 BLE 5.0 九轴无线姿态传感器模块，内置陀螺仪、加速度计、磁力计和 Cortex-M4 处理器，运行卡尔曼滤波算法，可以实时输出欧拉角、四元数、角速度等姿态数据。关键参数：
- 姿态精度：0.2°
- 数据率：默认 10Hz，最高 200Hz
- 重量：约 9g
- 蓝牙传输距离：50-90m（空旷）
- 续航：40 小时（100mAh 电池）

我们的方案是将 BS-BT91 封装在 3D 打印的足球外壳内，制作一颗**真智能足球**。在 VR 训练模式中：
- 球的旋转姿态由 BS-BT91 通过 BLE 实时传输 → Unity 中足球的 3D 模型同步旋转
- 球的**空间位置**暂由虚拟场景固定（训练场中心），因为 IMU 无法提供绝对定位
- 未来可通过 UWB 室内定位或 OpenCV 俯视摄像头补充位置信息
- 由于BS-BT91对于Windows适配问题，现阶段暂未实现联调功能

当前 P11 状态是**代码骨架已完成**（`ISmartBallSource` / `MockSmartBallSource` / `BleSmartBallSource` / `SmartBallController` / `TraininingModeController`），Mock 模式下可在 PC Editor 看到足球旋转效果。真机联调需要拿到 BS-BT91 的蓝牙数据帧格式文档。

### Q21：IMU 只有姿态没有位置，你们怎么解决足球定位问题？

**参考作答**：
分两步走：

**短期（演示阶段）**：球的 3D 位置由 Unity 虚拟场景固定（训练场中心点），只同步旋转姿态。在 VR 中这足以体现"数字孪生"的观感——你看到 VR 里的足球和桌上的真球转得一模一样。

**中期（下一阶段）**：三种备选定位方案：
1. **UWB 室内定位**（如 Decawave DWM1000）：在球场四角布置 Anchor，球内装 Tag，精度 ±10cm
2. **俯视摄像头 + OpenCV**：USB 摄像头从正上方拍摄球场，通过颜色/ArUco 标记追踪球的 XY 坐标，精度 ±2cm，成本最低
3. **多摄像头交叉覆盖**：2-3 个廉价 USB 摄像头三角测量，消除单目深度歧义

## 八、未来展望与扩展性

### Q：如果给你们更多时间，下一步做什么？

**参考作答**：

### Q：为什么不直接用现成的足球游戏引擎，要从头搭建？

**参考作答**：
现成的足球游戏（FIFA / eFootball / Rocket League）是**封闭的商业产品**，无法做到：
- **接入真实硬件**：没有开放接口接收 NetworkTables 数据
- **自定义剧本**：不支持教练自定义战术关键帧
- **VR 第一人称**：不支持 Quest 3S 的 6-DoF 沉浸观看
- **评分系统定制**：评分标准由游戏设计者写死，不能按训练目标调整
- **部署到 Quest**：商业游戏引擎的 APK 体积通常在 1-2GB，不适合 sideload 到 VR 一体机

Unity 作为通用引擎给我们的是**完全的控制权**——从数据输入到渲染输出，每一层都可以定制。而且 URP + XR Interaction Toolkit 的 VR 开发链是业界最成熟的免费方案。

## 九、比赛特定问题

### Q：萌芽赛道看重"创新性"，你们的创新点在哪？

**参考作答**：
三个核心创新点：

1. **虚实融合的交互范式**：不是纯虚拟仿真，也不是纯物理训练——真硬件触发 + 虚拟推演 + VR 沉浸观看，三者形成闭环。这在国内 FRC/机器人教育赛道是首创。
2. **可插拔的模块化架构**：数据源（FakeData ↔ NT）、AI 决策（预设剧本 ↔ 强化学习）、交互设备（PC ↔ Quest VR）都可以独立替换，不影响其他模块。这是工程思维的体现。
3. **训练工具的定位**：不是做一个"好玩的足球游戏"，而是做一个"有用的训练沙盒"。评分系统、慢动作回放、战术建议都是围绕训练目标设计的。

---

### Q：项目的技术难度体现在哪里？

**参考作答**：
- **跨领域集成**：机器人控制（WPILib C++）+ 3D 引擎（Unity C#）+ VR 部署（Android/Quest）+ BLE 通信（嵌入式），涉及 4 个技术栈的协同
- **事件驱动架构设计**：11 个模块通过 4 个事件松耦合通信，在高中生项目中比较少见
- **VR 性能优化**：在 Quest 3S 移动芯片上跑 URP 3D 场景 + 多角色 + 粒子特效 + 后处理，帧率从 45fps 优化到目标 72fps，需要大量的渲染管线调优
- **IMU 数字孪生**：将 9 轴传感器数据实时映射为 3D 模型的旋转，涉及四元数插值、低通滤波、坐标变换

### Q：如果评委说"你们这个不就是做了个动画吗？"，怎么回应？

"表面看像动画，但底层有三个关键区别：

### Q：项目还有什么不足？下一步打算怎么改进？

**参考作答**：
